# 发现 (Thou iOS)

## 2026-05-10: OpenClaw 真机首连预填的真实根因

### 1. 真机里“没有预填”不等于 UI 没绑定上
- 直接从设备容器拉回 `Library/Preferences/Zheng.Thou.plist` 后确认：当设置页只显示 `8080` 时，`OpenClawConnectionProfile` 里确实没有 `lastSuccessfulTarget`、`savedTargets`，`manualTarget.host` 也是空字符串。
- 这说明此前那次真机现象不是“有记忆但没显示”，而是设备侧根本没有可供回填的主机资料。

### 2. 从设备外部写 plist 不能作为稳定修复手段
- 本 session 试过把当前 Mac 的 host / token 直接写回真机 plist，再重启 Thou。
- 回读结果确认：App 启动后又把这些值覆盖回自己的旧空值。
- 结论：对 Thou 当前运行态而言，依赖设备外部改 plist 不能稳定改变 `UserDefaults` 域；这类手段只适合取证，不适合作为产品修复路径。

### 3. 当前可用修复是 app 内 bootstrap，而不是把 token 写进包里
- 已在 Thou app 内增加 OpenClaw bootstrap 主机来源：当 profile 与 legacy key 都为空时，手动地址会自动预填默认主机与端口，但仍允许用户编辑。
- 当前刻意不把完整鉴权 token 硬编码进源码或包内配置；token 仍然维持“手填一次后本地持久化”的策略。
- 这次真机验证已确认：bootstrap 主机预填生效，设置页现在是“主机预填、可编辑、token 为空待补”的状态。

### 4. 测试期可接受把 token 限定为 Debug-only 预填
- 用户已明确接受把“测试便利”和“正式产品行为”拆开：测试期允许 Debug 构建预填当前 bridge token，正式包仍不预装 token。
- 本轮已将 token 下沉到 Xcode Debug-only build setting，而不是公共 Info.plist 常量；真机已验证当前可达到“主机预填 + token 预填 + 可见清空口令”。

### 5. 当前更适合把 Thou 从桌面 `main` 会话里切出去，而不是新建第二个 channel
- 用户已接受新的收敛：Thou 不需要再起一个新的 OpenClaw channel，但当前默认共用桌面端 `main` 会话不利于 debug，也容易把桌面主会话噪音混进移动端判断。
- 当前更合理的方向是：仍使用同一个 Thou Native Plugin/channel，但把默认 conversation key 从共享 `main` 改成 Thou 自己的主会话 key。

### 6. OpenClaw iOS 的偶发卡死已收敛但未根治
- 当前真机现象不是“消息没发出去”，因为电脑端已能收到；更像是 iOS 在收到回复期间的 UI/状态更新链偶发卡住。
- 本轮已做两处本地修复：一是补上 `chat.done / error / disconnect` 的回合收口，二是把 WebSocket delegate 从主线程移出并去掉逐 chunk 日志。
- 用户最新反馈是“有改善，但仍偶发”，说明当前修复方向是对的，但还没有完全打到根因。

### 7. 真机 crash log 已确认 SwiftUI lazy layout 是这轮卡死的重要根因
- 后续继续拉取真机 crash / watchdog 日志后，主线程栈明确落在 SwiftUI 的 `LazyVStack`、`ScrollGeometry`、layout 路径，而不只是 OpenClaw 网络或流式状态层。
- 因此本轮没有继续猜测 WebSocket 或 session，而是直接把聊天列表从 lazy 堆叠改成普通 `VStack`，先避开这条真机崩溃路径。
- 最新真机安装并回归后，用户反馈已更新为“明显改善，基本不再卡死”；这说明这次修复命中了真实热点，但后续仍需关注 `RoundView`、自动滚动等相邻布局路径是否还存在余震。

### 8. Alice 当前不应再把 VPN/Tailscale 当作单一根因
- 用户新一轮实测确认：Alice 在 VPN 连接状态下，Wi-Fi 开和关两种情况下都出现过成功发送。
- 因此当前更准确的判断是：VPN/Tailscale 不是必要条件，而是放大器；它会提高异常出现概率，但不能解释全部失败与成功样本。

### 9. 插件握手资料此前过瘦，已经补上第一版 host hints
- 之前 native-plugin 在 `auth-ok` 里回给 iOS 的 mobile connection profile 只包含 `listenPort`、`listenPath` 和 `pairingTokenPrefix`，不会同步插件自己知道的可达 host 线索。
- 这会导致手机虽然能记住“本次刚好连上的地址”，但拿不到更多 LAN / Tailnet 候选，后续自动重连与设置页回填都偏弱。
- 本轮已把插件侧 mobile connection profile 扩成包含 `hostHints`，来源为当前监听 host、机器 hostname / `.local`、以及本机外网 IPv4 接口地址；iOS 握手后会把这些线索并入 `OpenClawConnectionProfile.savedTargets`。
- 当前验证结果：Swift 文件级错误检查通过，native-plugin `validate:session-routing` 通过，Thou 真机构建通过并已重新安装到设备。

## 2026-03-16: 协议握手与超时排查

### 1. 现象分析
- **日志**: `WebSocket 握手成功！` -> `Sending Auth Token...` -> `Received: connect.challenge` -> `-1001 The request timed out`.
- **结论**: 此 `-1001` 并非 TCP 连接超时 (因为已经收到了 Challenge)，而是 **应用层协议超时**。
- **原因**: OpenClaw 服务端发送 `connect.challenge` 后，期待客户端在短时间内响应该挑战。iOS 端可能阻塞在发送 Token 步骤，或者发送的格式不符合服务端期待，导致服务端主动关闭连接或客户端读取超时。
- **修正方向**: 检查 iOS 端对 `connect.challenge` 的自动响应逻辑，确保不阻塞主线程。

### 2. 网络连通性
- 确认 Mac 开启防火墙并不一定会完全阻断连接，但可能导致不稳定。
- 确认 `index.mjs` 代理模式 (Node Mode) 能成功接收 iOS 请求并转发 Packet 给 OpenClaw。

## 2026-03-19: Tailscale 接入方案骨架

### 1. 官方网络约束
- Gateway 是唯一长期持有 channel 连接与 WS control plane 的进程；一机建议一个 Gateway。
- Gateway 默认 loopback first，若要被 tailnet 访问，应显式使用 `gateway.bind: "tailnet"` 或 Tailscale Serve/Funnel 方案。
- Bonjour 只解决局域网发现；跨网络发现应改用 tailnet DNS / MagicDNS / Wide-Area Bonjour，或退回手动 host/port / SSH。

### 2. Thou 当前方案的落点
- Thou 不是要直接变成官方 iOS node，也不是现在就实现完整 Gateway operator 协议。
- 因此当前更自然的接法是：
    - iPhone 通过 LAN 或 Tailnet 到达 Mac 上的 sidecar。
    - sidecar 再连接同机 Gateway。

### 3. 双模式建议
- **LAN 模式**: 继续使用当前局域网 IP + 配对码。
- **Tailnet 模式**: 使用 Tailnet 地址 / MagicDNS 到达 Mac，不引入自建公网服务器。
- **Manual Host 模式**: 作为兜底，允许用户手输 host/port。

### 4. 下一步实现原则
- 先完成 Gateway WS chat 模型迁移，再做 Tailnet 地址发现。
- 不建议把网络接入改造和聊天协议改造混在同一次实现中。

## 2026-03-29: Alice 类记忆 Skill 的分层抽象

### 1. 记忆不是整包回放
- 对 Alice 的类记忆设计，不应理解成“把旧聊天记录整段塞回 prompt”。
- 更稳的模型是分层代谢：当前消息区保留最近对话，历史聊天负责可追溯，Topics 负责主题抽象，Impressions 负责下一轮 persona 注入。

### 2. retrieve 与 archive 不能混成一个动作
- retrieve 解决的是“当前轮需要想起什么”。
- archive 解决的是“这一轮结束后应该沉淀什么”。

### 2. DeepSeek thinking mode 暴露了一个重要约束：reasoning 的回传规则不是统一的上下文工程常识，而是 provider-specific 协议
- 用户关注的 DeepSeek 文档为：`https://api-docs.deepseek.com/zh-cn/guides/thinking_mode`。
- 其中一个关键差异是：
    1. 无工具调用时，上一轮 `reasoning_content` 在后续轮次里可以不回传，回传也会被忽略。
    2. 一旦该轮发生工具调用，则该轮的 `reasoning_content` 在后续所有请求中必须完整回传，否则 API 直接报 400。
- 这说明对多 provider / 多模型的 Thou 来说，`reasoning` 不能再被简单理解成“UI 展示用附属文本”；它有时还是 provider 协议状态的一部分。
- 因此后续若 Thou 支持更多 reasoning + tools 模型，需要把下面三件事拆开：
    1. 展示给用户的 thinking。
    2. 本地是否保存 reasoning。
    3. 某 provider 在后续请求里是否要求回传 reasoning。
- 这三者不能默认等价，也不能共享一条统一规则。

### 3. 对 Alice / Thou 的直接启发
- 当前 Thou 的 Alice 路线仍主要围绕 OpenRouter；但后续若考虑 DeepSeek 或其他 reasoning + tools provider，必须预留 provider-specific 的 message replay 策略。
- 更具体地说，未来的 provider 抽象层至少要允许每个 provider 自己定义：
    1. 哪些字段进入下一轮拼接。
    2. 哪些 reasoning 仅供展示。
    3. 哪些 reasoning 在工具调用后必须保留并回传。
- 否则后续一旦接入这类模型，统一消息模型很容易在“普通多轮”能跑、在“工具调用多轮”时报协议错误。

### 4. 对 Thou 当前实现应进一步收紧为 OpenRouter-first，而不是 provider-general
- 用户已明确补充：DeepSeek 这次讨论主要是满足个人好奇，不是 Thou 当前实现必须参考的协议来源；Thou 大概率只会围绕 OpenRouter 一家继续推进。
- 当前补读 OpenRouter 文档后，可以确认的点是：
    1. OpenRouter 统一了 reasoning 的控制参数，通过 `reasoning` 对象暴露 `effort`、`max_tokens`、`exclude`、`enabled` 等能力。
    2. OpenRouter 统一了 reasoning 的返回位置，返回在消息的 `reasoning` 字段中。
    3. OpenRouter 统一了 `tools` / `tool_choice` / `parallel_tool_calls` 的请求形状，并负责把 OpenAI 风格的工具调用请求转换给非 OpenAI provider。
- 但当前没有看到 OpenRouter 文档明确承诺“不同底层 provider 在工具调用后的 reasoning replay 差异，也全部由 OpenRouter 对客户端完全抹平”。
- 因此对 Thou 当前更稳的工程判断不是“现在就做 provider-specific replay 抽象”，而是：
    1. 先严格按 OpenRouter 文档的统一接口和返回字段实现。
    2. 不把 DeepSeek 的 reasoning_content 回传规则直接投影成 Thou 当前架构要求。
    3. 同时在概念上保留一条边界：`reasoning` 不是天然等于“纯 UI thinking 文本”，未来若 OpenRouter 某些模型/能力对 replay 有额外要求，再按 OpenRouter 文档更新实现。
- 这意味着当前 Alice 线可以继续维持 OpenRouter-first 的简化判断：
    1. system prompt、messages、tools 都按 OpenRouter 当前标准 shape 组织。
    2. reasoning 是否保存、展示、进入 archive，按产品需要决定。
    3. reasoning 是否必须进入下一轮拼接，当前只以 OpenRouter 实际文档与实测行为为准，不以其他 provider 文档做预设。

## 2026-04-20: 从长对话原型提炼 Alice Methodology

### 1. 资源是理解一个人的地基
- 基于 [Thou (iOS)/了解一个人.md](Thou%20(iOS)/%E4%BA%86%E8%A7%A3%E4%B8%80%E4%B8%AA%E4%BA%BA.md) 的长对话，可稳定提炼出一个核心判断：一个人为什么会是现在这个样子，首要前提不是人格标签，而是其拥有和可支配的资源。
- 这些资源包括但不限于金钱、时间、生活状态、资源来源、现实压力。
- 因此，Alice 的后续记忆与解读方法学中，资源信息不应只是背景资料，而应是解释其余行为和表达的背景滤镜。

### 2. Alice 不应问“有没有需求”，而应理解“如何满足”
- 长对话里一个稳定共识是：社交、认知、尊重、实现等需求并不是少数人才有，而是人人都有。
- 差异不在于有无，而在于人们用什么方式满足，以及为什么会选择这种方式。
- 这条判断直接影响 Thou：retrieve / impressions 不应轻易产出“用户没有社交需求”这类结论，而应产出“用户当前更常用何种满足策略”这类解释材料。

### 3. 虚拟满足和低成本替代不能被先验贬低
- 长对话多次强调：虚拟社交、单向连接、低成本信息消费、幻想式目标等，都可能是真实需求的替代满足方式。
- 这些方式当然可能带来代价，但不能被预设为低级或无效；应结合资源状态、时代环境和人格倾向一起解读。
- 这对 Alice 很重要，因为后续 archive / topic 提炼若直接按“高质量/低质量满足”做道德排序，会损失大量真实人物信息。

### 4. 矛盾性本身是高价值信号
- 长对话中最有价值的一部分，不是把人归类，而是识别其内在拉扯：行为、理想、资源、代价之间可能并不一致。
- 因此，后续 impressions 候选卡片应优先记录“反复出现的矛盾与补偿策略”，而不只是稳定特质标签。

### 5. Methodology B 应服务于“解释材料”生产
- 从这份原型长对话反推，Methodology B 的目标不应只是把旧聊天压缩成摘要，而应把长对话代谢成可复用的解释材料。
- 最值得沉淀的是：资源地基、需求满足策略、关键矛盾、长期收益与代价模式。
- 如果 archive 继续只以“最后一句话做标题和摘要”，retrieve 结果就会继续只能回忆出碎片，而解释不出“为什么会这样”。

### 6. 长对话后段补充了更清晰的“分层产物”结构
- 在 2000 行之后，原文不只是继续讨论问卷实现，还把最终产物拆成了更清晰的层级：三我画像、大五雷达图、标签云卡片。
- 这对 Thou 很重要，因为它说明“人物理解材料”并不只有一种粒度：
    - 三我是高层整合画像。
    - 大五是个性差异骨架。
    - 标签云是压缩后的展示层索引。
- 因此，不能把这三类东西都粗暴归入 impressions；否则运行时注入、归档沉淀、最终展示会混层。

### 7. 资源理论和大五不是互斥，而是共性与个性的分工
- 原文前半段强调资源地基，是在纠偏“唯内在人格解释论”。

- 原文最后形成的不只是问卷结果页设想，还形成了一套比较完整的提示词分层：世界观设定、角色扮演、输入信息、步骤化解读。

### 9. 标签云不是底层事实，而是上层压缩结果
- 原文里的“标签云卡片”来自多个模块解读摘要的二次压缩。
- 因此在 Thou 中，标签云更适合作为分享层或可视化层，而不是底层记忆事实源。
- 这条边界可以避免未来把为了展示而压缩出来的词，反过来当成运行时的长期记忆注入。

### 10. Methodology A / B 与 Impressions 的颗粒度不能过粗
- 不应把整份问卷结构直接等价成 Thou 的 Methodology A 或 Methodology B。
- 对 Thou 更有价值的是提炼出高层原则、观察焦点、归档边界，而不是照搬整套题面。
- 更合适的分工是：
    - Methodology A 负责当前轮的理解焦点。
    - Impressions / Topics 负责当前已沉淀出的中层理解材料。
    - Methodology B 负责一轮结束后如何把交流继续代谢成新的中层材料。

## 2026-04-18: Alice 真实 archive / session 调试补记

### 1. 当前 Alice 还没有真实“前台可调用工具”
- 从真实会话文件可见，Alice 会在对话正文里说“【工具调用：存档对话】”“【工具调用：调取记忆】”，但同期模拟器 `ThouMemory` 中并没有对应的 `Messages/`、`Topics/` 或 `Impressions` 更新。

### 2. archive 曾存在双重裁剪 bug
- `AliceMemoryOrchestrator.archiveAfterResponse(...)` 早先会先把 `rounds.dropLast(keepRecentRounds)` 当作 `ArchiveRequest.rounds` 传下去。
- `MemorySkillService.archive(_:)` 收到后又会再做一次 `dropLast(keepRecentRounds)`。
- 结果是：理论上命中了 archive 阈值，也可能因为第二次裁剪把待归档数组裁空，导致 `Messages/` 不落真实文件。

### 3. session 持久化与 active rounds 是两层事实
- `Sessions/*.json` 里的 `rounds` 是完整会话旁路，用于恢复和排查；`activeRoundIDs` 才是当前真正参与前台展示与下一轮上下文的活跃窗口。
- 因此“完整历史已经存在”不等于“当前轮一定还能看到全部历史”；active 窗口被截断后，如果 retrieve 又没补回，用户会感受到明显的“失忆”。

### 4. 当前前台提示先走极简态，不展开完整工具流水
- 对 Alice 这条线，现阶段更合适的 UI 不是把每次后台 archive / retrieve 的入参、出参、执行细节全部显式打印出来。
- 第一版先只暴露两个用户可感知状态：retrieve 尝试时显示“正在回忆”，archive 真正成功后显示“已归档”。
- 这能先验证交互方向，而不把调试视图提前做得过重。

### 5. 必须能一键洗掉 Alice 测试环境
- 由于 Alice 会把完整 session 旁路持久化到 `ThouMemory/Sessions/`，如果没有显式重置入口，重装/重建后仍可能因为同一个 session 锚点继续恢复旧测试历史。
- 因此需要同时清文件和重置 `AliceCurrentSessionID`，仅删 `ThouMemory` 目录还不够彻底。
- `rounds`: 本次准备归档的历史前缀轮次。
- `keepRecentRounds`: 前台保留窗口，只影响前台代谢与下一轮上下文，不影响 archive 文件里实际写入的归档轮次。
- `topicUpdates`: 本轮要续写或创建的 topic 列表；第一版已允许一次 archive 对多个 topic update 建模，虽然当前启发式默认只落一个 focal topic 或首个 retrieve topic。
- `impressionUpdates`: 慢变量印象更新列表，和 topic 更新并行写入。
- archive 输出通过 `ArchiveResult` 返回 `archivedRoundIDs`、`keepRecentRounds`、`touchedTopicIDs`、`impressionsUpdated`，供 orchestrator 再映射成前台 active rounds。

### 4. 当前 Topics / Impressions 边界

- Impressions: 只接稳定偏好、长期状态、计划倾向等慢变量；第一版启发式只从用户文本里抽带有“喜欢 / 不喜欢 / 计划 / 一直 / 最近 / 希望”等标记的句子，并限制最多更新 3 条。
- `keepRecentRounds` 只负责前台上下文窗口，不负责 topic 事实筛选，也不负责 impressions 的更新开关。

## 2026-04-18: Alice 前台失忆感与完整会话持久化

- Alice 联调卡现在不只显示 retrieve / reasoning / 正文。

- 当前是否处于“活跃窗口已小于完整会话”的截断状态。
- 当前显示轮次 vs 完整会话已记录轮次。
- 最近一次 archive 的摘要。

## 2026-04-09: Alice 401 `Missing Authentication header` 的判定基准

### 1. 官方端点本身没有变成别的 host
- OpenRouter 官方文档当前仍明确写 `https://openrouter.ai/api/v1` 作为 API base URL，`/api/v1/chat/completions` 仍是有效 POST 端点。

### 2. `Missing Authentication header` 与 `User not found` 代表两种完全不同的问题
- 对同一端点直接用终端发送一个带假 Bearer 的 POST，请求会得到 `401 User not found.`
- 这说明只要服务端真正收到了 `Authorization`，即便 key 是错的，也不会回 `Missing Authentication header`。
- 因此 Alice 里出现 `Missing Authentication header` 时，优先怀疑的是“iOS 实际发出的请求没有带出合法 Authorization header”，而不是简单地继续怀疑 key 填错。

### 3. 当前客户端侧补强策略
- 已把 Authorization 同时写到 `URLRequest` 和 `URLSessionConfiguration.httpAdditionalHeaders`。
- 已增加脱敏诊断摘要：只显示 key 长度、前缀/后缀和是否含空白/控制符/非 ASCII，不显示完整 key。
- 后续若仍报 401，可先看联调卡中的 key 来源与摘要，再决定是继续查运行实例状态，还是继续查 URLSession 层行为。

## 2026-04-12: Alice 401 的真实根因不是 Header 丢失，而是本地旧配置污染

### 1. 最终定位结果
- `AliceLocalSecrets.plist` 中保存的是正确的 `sk-or-...` key。
- 运行时联调卡里出现的却是 `so-or-...` 前缀；继续排查后确认，这个错误值来自模拟器中 Thou 的 `UserDefaults`，而不是源码或打包资源。
- 由于 Alice 旧逻辑是“优先读 UserDefaults，只有为空才回退到 plist”，因此这条错误旧值持续覆盖了正确 key。

### 2. 这说明什么
- 之前的 `401 Missing Authentication header` 在 Alice 这条链上并不是 OpenRouter 端点变化，也不是请求头发送机制天然失效。
- 更准确地说，是一条拼错前缀的旧 key 让排查表象看起来像“header 没带出去”。
- 本地 App 联调里，历史持久化状态本身就是一级排障对象，不能只盯代码与接口文档。

### 3. 已采取的收口
- Alice 配置层现在会对持久化 key 做最小形态校验；若 `UserDefaults` 中的值明显不像 OpenRouter key，则自动忽略并回退到 `AliceLocalSecrets.plist`。
- 因此后续再看到 Alice 401，第一反应不应再是“OpenRouter 是不是又改了”，而应先看联调卡里的 key 来源与脱敏摘要。

### 4. 当前 Alice 主线已经切换
- Alice 真实 OpenRouter 请求链已经跑通。
- 当前下一步不该继续停留在连通性排障，而应回到记忆 skill 主线：archive 接线、`keepRecentRounds`、Topics / Impressions 更新策略。

## 2026-03-31: 为什么每次连接都会新增一条 OpenClaw 会话

### 1. 直接根因
- Thou iOS 客户端原先发送 `auth` 时只带 `token`，没有带稳定的 `deviceId`。
- Thou bridge 在缺少 `deviceId` 时会为每次新连接生成一个新的 `device:uuid`，于是宿主侧 session key 也会随之变成新的 `agent:main:thou-ios-device:*`。

### 2. 已采取的修正
- 当前 `ClawConnectionManager.swift` 已把 `deviceId` 持久化到 `UserDefaults`，并在 `auth` 时同时发送 `deviceId` 与 `deviceLabel`。
- 这意味着同一台 Thou 设备后续重连时，宿主应看到同一个 Thou 设备身份，而不是每次一个新的随机设备。

### 3. 这和“未来能否选择之前的会话”是什么关系
- 先要解决“同一设备身份稳定”这个前提，否则会话选择 UI 会被一堆看似不同、实则同源的随机 session 污染。
- 设备身份稳定后，再评估在 Thou 的 OpenClaw 页面中增加“继续当前会话 / 选择历史会话”才有意义。

## 2026-05-09: Alice 真机 OpenRouter 500 的最新判断

### 1. 当前已确认和已排除的边界
- 这轮没有改动 Alice 的记忆编排文件或记忆存储文件；当前留下的 Alice 改动集中在 `OpenRouterClient.swift` 和 `AliceSettingsView.swift`。

- 之前的判断“关掉 VPN 后恢复”只够说明 Tailscale / VPN 是强放大因素，不够说明它是唯一根因。

- 因此后续排障不应再把问题二分成“开 VPN 就坏、关 VPN 就好”，而要继续采样关闭 VPN 后的失败样本。

### 3. 当前代码侧采取的收敛动作
- `OpenRouterClient.swift` 现已对 HTTP 500 增加有限次重试，并在流式失败后继续尝试非流式兜底。
- 保留请求诊断字段仍然有价值，因为现在要比较的是“同一 payload 在不同真机出口 / 不同尝试轮次下的结果差异”，而不是只排查 header 或 key 形态。

## 2026-05-10: Alice 真机 500 的阶段决策与边界收紧

### 1. 当前阶段决策
- Alice 传输层当前先冻结，不回退到更早的请求实现。
- 这个决策不是因为现状已经解决，而是因为“模拟器正常、真机异常、VPN 开启时失败概率显著更高”已经说明当前最大变量更像是真机网络出口，而不是单纯请求 JSON 形态。
- 在没有新的真机失败样本前，回退传输层只会同时改变 payload、重试和诊断面，扩大变量面，不利于继续定位。

### 2. 关于“如果没有 Tailscale / 没有 OpenClaw 页面，问题还会不会存在”的思想实验
- 这个思想实验有价值，但当前只能得出“问题类别大概率会缩小到手机侧网络路径 / VPN 网络扩展层”，还不能把它当成严格证明。
- 更准确的说法是：如果 Thou 变成只有 Alice、且手机上也没有 Tailscale 参与网络出口，那么“由 Tailscale iOS VPN/Network Extension 干扰 Alice 外网请求”这一类问题大概率会消失。
- 但这仍不能自动推出“当前全部问题都只由 Tailscale 造成”，因为真机环境里还可能存在其它系统网络因素、运营商路径、Wi-Fi/蜂窝切换、以及上游服务端对移动出口的偶发处理差异。
- 因此当前最佳表述应是：Tailscale / VPN 已被确认是强放大因素，而不是唯一已证实根因。

### 3. 关于 OpenRouter 错误面的后续排障锚点

## 2026-04-02: 第三刀为什么要继续拆成模块 ViewModel

### 1. 第二刀之后还剩的核心问题

- Alice 现在有自己的模块 ViewModel，后续可以直接围绕它继续扩 system prompt、model 选择、retrieve / archive。
- OpenClaw 现在也有自己的模块 ViewModel，后续可以围绕它继续扩连接状态机、session 操作、历史回放和远程聊天。
- 总 `ChatViewModel` 现在更接近真正的 App Shell 协调器，而不是“把所有业务都塞进去的大桶”。

### 3. 这对后续 session 的意义

- 因此在没有先核对 Simulator 设置前，不建议继续强攻输入框代码本体。
- 后续对照验证结果：在同一台模拟器里的系统 Reminders 输入框也复现了相同现象，因此这条判断基本坐实。Thou 输入框问题暂记为外部环境问题，后续等 Xcode / Simulator 环境稳定后再回头处理。

## 2026-04-01: 真实会话选择后端的最小接法

### 1. 为什么不让 iOS 直接去连 Gateway
- Thou 当前已经稳定依赖 native-plugin bridge 做 auth、chat 和流式回包。
- 在 iOS 端再单独接一套 Gateway 协议，只会把鉴权、会话解析和错误处理再复制一遍，不划算。
- 因此本轮选择的是：继续把 bridge 当成唯一入口，但把 `sessions.list`、`sessions.create` 和 `sessionKey` 透传能力补进去。

### 2. 当前真实会话选择的边界
- 现在的会话选择已经不再是假的：列表来自宿主 `sessions.list`，新会话来自宿主 `sessions.create`，后续消息发送会带真实 `sessionKey`。
- 这批改动的目标是先把“选择哪个真实会话发消息”这件事做通，而不是一次性补完完整 transcript 历史回放。
- 如果后面需要在切换时直接回显历史消息，可以继续接 `sessions.preview` 或更完整的 transcript 读取接口，但这不是这轮的阻塞项。

### 3. 假会话入口的当前定位
- OpenClaw 页左上角现在有一个假的会话按钮，会弹出“新会话 + mock 历史会话”的前端占位列表。
- 它当前只负责把会话选择 UI 的位置、交互和视觉壳子固定下来，不负责真实历史会话读写。
- 后续接真实会话后端时，应尽量只替换数据源和行为，不再重做这层前端结构。

### 2. 第一版最合适的新增层是编排器，不是大而全 memory manager
- 对 Thou App 来说，最自然的新增对象是 `AliceMemoryOrchestrator` 一类的编排层。
- 它位于 `ChatViewModel` 和 `OpenRouterClient` 之间，负责两件事：
    - 发送前执行 retrieve，并组装 system prompt / impressions / topics / active messages。
    - 回复后执行 archive，并把 `keepRecentRounds` 返回给前台。
- 这样可以保持 `OpenRouterClient` 继续只做网络流式请求，避免把记忆语义压进网络层。

### 3. retrieve 与 archive 的边界已明确
- retrieve 是在线动作，发生在发送前，目标是“当前轮需要想起什么”。
- archive 是离线化动作，发生在 assistant 回复完成后，目标是“这一段结束后该沉淀什么”。
- 第一版不应在流式生成中途触发 archive，否则 UI、并发和调试成本都会升高。

### 4. Impressions 不能直接从 messages 生写
- 最新边界已修正：Impressions 应在 archive 流程中与 Topics 同步更新。
- 它的输入依据应同时参考原始消息与本轮 topic 更新结果，而不是被限制为只能从 Topics 二次归纳。

### 5. 第一版的最小验证目标
- 能在 Alice 请求前注入短的 impressions 和少量 retrieved topics。
- 能在回复完成后得到一个 `keepRecentRounds` 决策，并让前台 `rounds` 代谢截断。
- 只要这两个动作闭环成立，retrieve / archive / topics / impressions 四层就算完成了第一版可执行落地。
