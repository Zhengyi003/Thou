# Alice 分层原型

## 目的

这一版不是问卷还原，也不是最终数据模型，而是先把 Alice 的四类材料切层：

1. 哪些属于运行时理解层。
2. 哪些只属于归档与代谢层。
3. 哪些适合作为中层可持久化材料。
4. 三我、大五、标签云分别处于哪一层。

## 重要纠偏：运行时不是存储层，而是可见性阶段

上一版里把“运行时理解层”写得太像一个独立存储层，这个表述容易误导。

更准确的说法是：

- L0 到 L4 先描述材料的语义分层与职责分层。
- 运行时描述的是“某一轮请求发给模型时，哪些层的哪些材料会被装配进上下文，因此被模型看见”。

也就是说：

- 某个材料可以语义上属于 L1。
- 但在当前轮，它完全可以进入运行时，被系统提示词或检索块注入给模型。

例如：

- Impressions 语义上属于持久化理解材料。
- 但在聊天请求时，它可以完整进入运行时系统提示词。

因此后文的 L0 到 L4，应该理解为“材料层”。

## 总体分层

### L0. 产品宪法层

面向“无论聊什么，Alice 都应始终成立的产品身份与工作原则”。

这一层应包含：

- 产品定位。
- Alice 的角色边界与关系边界。
- 基础语气与互动原则。
- 完整的 Methodology A。

这一层通常每轮都可见，适合进入 system prompt。

### L1. 用户规范理解层

面向“Alice 当前对这个用户已经形成的、最新且完整的理解底板”。

这一层的核心不是原始证据，而是当前有效的 canonical user model。

这一层应包含：

- 最新且完整的 Impressions。
- 三我的结构化槽位。
- 大五的结构化槽位。
- 标签云的结构化槽位或语义索引槽位。
- 必要的人格骨架观察与关系骨架观察。
- 少量跨 topic 的稳定判断。

这一层不应包含：

- 原始聊天大段证据。
- 每条判断的全部证据链。
- 标签云。
- 三我整合长文。

这一层通常也适合每轮可见，进入 system prompt 或 developer prompt 的用户模型部分。

这里需要特别说明：

- 如果三我、大五、标签云在 Thou 里承担的是“用户认知框架”的一部分，那么它们的概念槽位就应属于 L1。
- 真正属于更高展示层的，不是这些概念本身，而是它们被渲染成结果页、卡片、雷达图、长文解释之后的表现形态。

### L2. 可检索记忆层

面向“哪些材料值得跨 Session 保留，并在未来被复用”。

这一层是 Alice 真正的长期记忆工作库，既不是当前 canonical user model，也不是最终展示结果。

这一层应包含：

- Topics：主题级、人物级、事件级、计划级的可检索材料。
- 原始 Session archive。
- 更细的事实片段与来源索引。

这一层的特点：

- 可追溯到来源。
- 可逐步修正。
- 可被 retrieve。
- 不要求直接给用户看。

这一层通常不是每轮默认全量可见，而是在命中检索后部分进入运行时。

### L3. 归档与代谢层

面向“这一轮结束后，系统如何把对话转成可复用材料”。

这一层的主体是 Methodology B，而不是用户可见内容。

这一层应包含：

- ArchiveDecision：是否归档、保留多少活跃轮次。
- 候选 Topic 提取。
- 候选 Impression 提取。
- 候选 Trait Observation 提取。
- 证据链、置信度、冲突标记、时间衰减规则。
- 派生结果的更新规则，而不是派生结果本身。

这一层通常不进入聊天请求本身，但会在 archive / summarize / update memory 这类后台子调用中被模型看见。

### L4. 整合与展示层

面向“如何把长期理解压缩成更高层画像，或可供用户查看/分享的结果”。

这一层应包含：

- 三我的结果页表达。
- 大五的雷达图、结果卡或长文解释。
- 标签云的展示形态。
- 其他总结卡片、结果页、分享图。

这一层是从 L1 和 L2 派生出来的结果表达，不应倒过来充当底层事实源。

这一层默认不进入聊天运行时，只有当用户明确要求查看、解释、分享这些结果时，才临时进入上下文。

## 上下文工程里的可见性关系

如果从上下文工程看，真正重要的不是“材料属于哪层”，而是“当前这次模型调用能看到哪几层”。

### 聊天主调用

聊天主调用里，模型通常会看到：

- L0：完整产品定位与完整 Methodology A。
- L1：最新且完整的 Impressions。
- L2：当前轮检索命中的 Topics / facts 子集。
- 当前活跃消息窗口。

聊天主调用里，模型通常看不到：

- L2 的全量 archive。
- L3 的归档规则细节。
- L4 的完整展示产物，除非本轮任务就是讨论它们。

### 检索调用

如果存在独立 retrieve 子调用，检索模块通常会看到：

- 当前用户输入。
- 最近消息窗口。
- L1 的少量索引性判断。
- L2 的检索库。

它的职责不是回复用户，而是决定把 L2 的哪部分送回聊天主调用。

### 归档调用

在 assistant 回复结束后的 archive 调用里，模型通常会看到：

- 刚完成的对话片段。
- 现有的 L1 canonical impressions。
- 相关的 L2 topics / facts。
- L3 的完整 Methodology B 与更新规则。

这个阶段不需要 L0 的完整产品对话语气，也不需要 L4 的展示卡片。

### 展示调用

当用户主动要求“给我看三我画像 / 大五 / 标签云”时，模型才会看到：

- L1 的用户模型。
- L2 的相关证据材料。
- L4 的生成模板或既有结果。

## 对上一版一个关键判断的修正

上一版写成“进入运行时的，应该是理解坐标和少量翻译成人话的稳定判断”，这个说法过窄。

更准确的版本应该是：

- 运行时应该看到完整的产品定位。
- 运行时应该看到完整的 Methodology A。
- 运行时应该看到最新且完整的 Impressions，但这份完整 Impressions 应该是 canonical summary，而不是原始证据全集。
- 运行时按需再看到 L2 检索出来的 topic 片段。

这里真正不应该默认常驻注入的，不是完整 Methodology A，也不一定是完整 Impressions；而是：

- L2 的全量 topics 与全量 archive。
- L3 的代谢规则。
- L4 的三我长文、大五结果页、标签云。

这里再补一条更准确的区分：

- 三我、大五、标签云如果作为 Impressions 的结构化字段存在，那么它们可以属于 L1。
- 但三我长文解释、大五结果页、标签云卡片这些“呈现结果”，仍然属于 L4。

## Methodology A

## 定位

Methodology A 是 Alice 的运行时理解方法，不是问卷，也不是存档器。

它回答的是：当前这句话、这个话题、这段互动，Alice 应优先从哪些角度理解。

## 建议职责

Methodology A 运行时优先看四类信号：

1. 资源地基。
2. 需求满足策略。
3. 反复出现的矛盾与补偿。
4. 当前关系语气与互动诉求。

## 建议结构

Methodology A 可先收敛成下面这组运行时观察轴：

- 资源地基轴：时间、金钱、精力、稳定性、外部支持、现实压力。
- 满足策略轴：用户通常通过什么方式满足社交、认知、尊重、实现等需求。
- 代价与收益轴：当前策略短期带来什么，长期牺牲什么。
- 矛盾轴：想要的、相信的、正在做的是否一致。
- 关系轴：这一轮用户更需要陪伴、理解、提醒、分析、还是推进。

## 运行时注入边界

Methodology A 应以规则和观察焦点存在，而不是以大量术语存在。

适合进入运行时的，是像下面这种短材料：

- 用户当前资源较紧，因此很多选择首先受现实约束。
- 用户常用低压力、间接式方式满足社交需求。
- 用户经常在理想自我与当前行动之间出现拉扯。

不适合进入运行时常驻注入的，是像下面这种材料：

- 大段理论说明。
- 完整人格分析报告。
- 三我长文。
- 标签云词表。
- 归档规则本身。

## Methodology B

## 定位

Methodology B 是 Alice 的归档与代谢方法，不负责当轮回复，而负责把对话转成可持续材料。

它回答的是：这轮聊完之后，哪些内容值得留下，应该以什么粒度留下，留下到哪一层。

## 建议职责

Methodology B 应重点识别五类产物：

1. 可复用事实。
2. 可复用主题。
3. 慢变量印象。
4. 人格/风格骨架线索。
5. 高层整合画像候选。

## 建议判断顺序

1. 这轮有没有新的事实、变化、承诺、反复出现的模式。
2. 这些内容属于某个已有 Topic，还是需要新建 Topic。
3. 这些内容是局部主题材料，还是已经影响到跨主题理解。
4. 这些内容是否只是短期情绪，不该升级成 Impression。
5. 这些内容是否只适合作为展示层派生结果，而不应反灌运行时。

## 归档输出边界

Methodology B 产出的不应是“最终人格结论”，而应是带来源的候选材料。

例如：

- 一个新 Topic：最近在回避高压力社交，但并未失去社交需求。
- 一条 Impression：遇到高成本现实约束时，更倾向于选择低风险替代满足。
- 一条 Trait Observation：外向性未必高，但对关系连接仍有稳定需求。

## Impressions

## 定位

Impressions 是 L1 中层可持久化材料里的“慢变量理解层”。

它不是聊天摘要，也不是展示页文案，而是 Alice 下次进入对话前对这个人的默认理解底板。

如果从 Thou 的产品心智看，Impressions 更接近“当前版本的用户模型”。

这意味着：

- 它可以有明确的结构化栏目。
- 某些栏目即便暂时为空，也不应缺席。
- 空缺本身就是一种信息，代表“这个维度尚未形成判断，后续应继续观察”。

## 应保留的内容

- 资源地基的稳定判断。
- 反复出现的满足策略。
- 反复出现的矛盾与补偿机制。
- 稳定的关系偏好与互动偏好。
- 低频但稳定的人格骨架观察。
- 三我相关槽位。
- 大五相关槽位。
- 必要的标签索引槽位。

## 不应混入的内容

- 单次聊天的情绪起伏。
- 原始聊天大段原文。
- 为展示而写的文艺化总结。
- 标签式堆砌。

## 推荐颗粒度

每条 Impression 都应满足：

- 可以被未来多轮对话验证或修正。
- 可以追溯到一个或多个 Topic / Session。
- 可以被改写，但不该因单轮对话剧烈翻转。

更具体地说，Impressions 不应只是“抽象判断句”，而应是“抽象判断 + 证据锚点”。

例如：

- 判断：她更偏保守。
- 简短解释：在关系推进、消费决策和公开表达上，反复优先选择低风险路径。
- evidenceTopicIDs: [topic-12, topic-27, topic-44]

这样做的意义不是把全部具象内容常驻注入，而是让模型在运行时先拿到抽象判断；一旦当前对话需要追溯，再顺着 evidenceTopicIDs 精确调取对应具象材料。

## 建议结构

Impressions 更适合长成“规范版用户模型”，而不是零散句子堆。

第一版可以先按下面的形态理解：

```text
ImpressionRecord
- profileVersion
- resourceFoundation
- needFulfillmentPatterns
- recurringTensions
- relationalStyle
- threeSelf
	- idealSelf
	- perceivedSelf
	- currentSelf
	- harmonyNotes
- bigFive
	- openness
	- conscientiousness
	- extraversion
	- agreeableness
	- neuroticism
- tagIndex
- stableJudgments[]
	- statement
	- rationale
	- evidenceTopicIDs[]
	- confidence
	- lastReviewedAt
```

其中：

- `threeSelf` 和 `bigFive` 可以允许空值或未定状态。
- `tagIndex` 更适合作为索引槽位，而不是最终展示词云本体。
- `stableJudgments` 用于承载那些还未完全适合塞进固定栏目、但已经足够稳定的判断。

## Topics

## 定位

Topics 是 L1 中层可持久化材料里的“可检索主题单元”。

它承担的是按需回忆，不承担默认人格解释总表。

它和 Impressions 的关系更接近：

- Topics 保留具象材料和主题脉络。
- Impressions 保留对这些具象材料做出的当前抽象理解。
- Impressions 中的重要抽象判断，应该能反查到 Topics。

## 应保留的内容

- 某个稳定主题的关键事实。
- 某个计划/人物/关系/兴趣的延续脉络。
- 与该主题相关的满足策略、代价、矛盾。
- 尚未解决的问题或未来可继续追问的缺口。

## 不应承担的内容

- 通用人格总评。
- 最终展示文案。
- 全局统一身份画像。

## 推荐颗粒度

一个 Topic 最好只解决一个可检索焦点，例如：

- 某类社交替代满足方式。
- 某段关系中的拉扯。
- 某个长期计划的推进与反复。
- 某个持续资源压力源。

## 三我 / 大五 / 标签云 的层级归属

### 三我

三我如果只是“结果页里的三我画像”，更适合放在 L4。

但如果三我是 Thou 的产品定位组成部分，是 Alice 理解用户的核心框架之一，那么三我的结构化槽位就应进入 L1 的 Impressions。

原因：

- 它既是高层文化表达，也可能是当前用户模型的一部分。
- 它需要汇总多个 Topic 和 Impression 才能逐渐成形。
- 它可以在 L1 以空槽位或半成形判断存在，在 L4 以长文画像存在。

运行时边界：

- 若三我属于 Alice 的核心理解框架，则其结构化槽位可以常驻运行时。
- 但不建议把完整三我长文画像常驻注入。

### 大五

大五更适合处于 L1 与 L4 之间：

- 底层上，它可以作为 L1 的 personality skeleton 和 Impression 槽位被低频持久化。
- 表达上，它可以在 L4 被组织成雷达图或结果卡片。

运行时边界：

- 如果大五槽位是用户模型的一部分，那么带空值的完整字段可以常驻。
- 但不建议把大五结果页说明文或过多术语化解释常驻注入。
- 更适合进入运行时的，是字段本身加少量人话解释。

### 标签云

标签云的展示结果应明确放在 L4。

但如果标签云在 Thou 里承担“认知索引”的作用，那么其底层 tagIndex 可以属于 L1 的一个辅助字段。

原因：

- 词云本体是压缩后的展示索引，不是事实。
- 但少量标签索引可以帮助模型快速扫描用户模型中的空位与重点。
- 需要避免把最终展示措辞直接当成底层事实。

运行时边界：

- 不建议把最终词云卡片常驻注入。
- 若保留 tagIndex，宜作为辅助索引而不是主叙述主体。
- 它不能替代 Impressions，也不能替代 Topics。

## 四个问题的直接回答

### 1. 哪些内容应该进入运行时理解层

- Methodology A 的观察坐标。
- 少量稳定 Impressions。
- 少量按需 retrieve 的 Topics。
- 当前活跃消息窗口。
- 极短的人格/关系倾向翻译语句，但不是术语表。

### 2. 哪些内容只属于归档与代谢层

- Methodology B。
- ArchiveDecision 与归档阈值。
- Topic / Impression 候选提取逻辑。
- 置信度、证据链、冲突处理、衰减规则。
- 各类派生结果的生成规则。

### 3. 哪些内容可以作为中层可持久化材料

- Topics。
- Impressions。
- 低频 Trait Observations。
- 必要的来源索引、时间戳、置信度。

### 4. 三我、大五、标签云分别处于哪一层，哪些不应进入运行时常驻注入

- 三我：主要在 L3，高层整合画像；不应完整常驻注入。
- 大五：骨架数据在 L1，结果可视化在 L3；不应把完整分数表常驻注入。
- 标签云：纯 L3 展示层；不应进入运行时，也不应作为底层记忆事实源。

## 当前推荐的最小落地方式

第一版先按下面的心智推进：

1. Methodology A 只定义运行时观察轴，不碰问卷细节。
2. Methodology B 只定义归档判断与提取边界，不急着做复杂评分。
3. Impressions 和 Topics 作为正式中层材料继续打磨。
4. 三我、大五、标签云先视为派生层，不反向主导运行时。
