# Thou (iOS) 进度追踪

## 当前主线

Thou 当前有两条并行线：

1. **Alice agent 线**：继续把 Thou 做成带代谢式记忆的移动端 agent。
2. **OpenClaw companion 线**：不是复刻 OpenClaw Web UI，而是压缩其移动端接入复杂度。

## Alice 线现状

- 已完成：
  1. Alice 配置层、请求层、ViewModel 边界初步拆分。
  2. `AliceMemoryOrchestrator` 已接入发送前 retrieve。
  3. reasoning 参数与流式解析已对齐当前 OpenRouter 形态。
  4. Alice 真实 OpenRouter 请求链已跑通，联调卡可观测到 key 来源、脱敏摘要、retrieve / reasoning / 正文命中情况。
  5. 已确认此前 `401 Missing Authentication header` 的真实根因是模拟器内 `UserDefaults` 残留的错误 key 覆盖了 `AliceLocalSecrets.plist`，不是 OpenRouter 端点或 header 机制本身异常。
  6. Alice 配置层已加入兜底：若本地持久化 key 形态可疑，则自动回退到 `AliceLocalSecrets.plist`，避免旧值继续污染运行实例。
  7. archive 已真实接入 Alice 回复完成后的发送链：assistant 流式完成后，会进入 `AliceMemoryOrchestrator.archiveAfterResponse(...)`，再由编排层统一决定是否归档、更新 topics / impressions，并把前台 `rounds` 截断回写。
  8. archive 输入已从“单 topic 写入”扩成“单次 archive + 多个 `ArchiveTopicUpdate` + impressionUpdates”，避免把 retrieve 命中的 topic 续写、focal topic 新建、impressions 更新三件事混成一个字段。
  9. 第一版前台代谢策略已落地：当前默认 `rounds >= 8` 才触发 archive，触发后前台保留最近 4 轮，其余轮次写入 `HistoricalChatArchive`。
  10. 已新增 Alice 完整会话持久化：除前台 `rounds` 活跃窗口外，`conversationHistoryRounds` 会以独立 session 文件落到 `ThouMemory/Sessions/`，用于后续排查“是否发生过存取技能、是否发生过前台截断”。
  11. Alice 聊天页联调卡已新增 archive / 截断 / 完整轮次 / 当前显示轮次信号，可直接观察“是不是只剩后几轮”和“是否刚发生过 archive”。
  12. 已修正 archive 的双重裁剪：`AliceMemoryOrchestrator` 不再先裁一遍再交给 `MemorySkillService` 二次裁剪，避免命中阈值后 `Messages/` 仍落不出真实 archive 文件。
  13. 已给 Alice 方法学补上约束：当前没有真实前台工具调用结果时，不得伪称“已调用 archive / retrieve / 存档 / 调取”。
  14. 已新增 Alice 测试环境重置入口：设置页可一键清空 `Sessions / Messages / Topics / Impressions` 并重置本地 session 锚点，避免旧测试历史继续污染新一轮验证。
  15. 已新增极简记忆态提示：retrieve 尝试阶段显示“正在回忆”，真实 archive 成功后显示“已归档”，暂不展开完整工具调用/返回明细。
  16. 已从 [Thou (iOS)/了解一个人.md](Thou%20(iOS)/%E4%BA%86%E8%A7%A3%E4%B8%80%E4%B8%AA%E4%BA%BA.md) 中提炼出 Methodology A / B 的第一版方向：Alice 的理解模型不再只盯人格标签，而是转向“资源地基 + 内在特质 + 需求满足策略”的组合框架。
  17. 已补充长对话后段的分层产物结构：三我作为高层整合画像、大五作为个性骨架、标签云作为压缩展示层，不再把它们统称为同一种 impressions。
  18. 已根据用户补充重新收紧措辞：马斯洛 / 大五被明确为现成的信息收集与观察方法，三我被明确为承载产品文化的新理论，标签云被明确为工具化展示层，不再和 Methodology / Impressions 混写。
  19. 已确认当前阶段先不继续做“从长对话中反向摘出完整问卷、再提炼成 Methodology A / B 最细颗粒”这条分枝；下一步先基于现有理解直接出一版可运行原型，等产品其他部分继续打磨时再回头细化这一层。

  21. 已确认 Alice 当前不回退传输层：在没有新真机样本前，先冻结请求实现、重试与诊断面，避免继续扩大变量面。

## OpenClaw 线现状

- 已完成：
  1. 局域网 bridge 主链已跑通，iOS 可完成鉴权、发送、流式回复与历史同步。
  2. OpenClaw 页面已移除显式新会话 / 历史会话入口，默认向单主会话体验收敛。
  3. 设置页已具备 `神奇配对码` 与 `手动地址` 两种连接入口。
  4. 已完成“连接我的 Mac”第一版连接资料概念：用一份可扩展资料同时承载局域网快捷入口、已知远程地址与鉴权信息，而不是把 LAN / Tailnet / 手动地址拆成三套产品心智。
  5. 已放宽“手动地址”输入：当前可直接接受 `裸 host`、`host:port`、`ws://host:port/thou` 等粘贴值，减少 Tailscale 联调前的地址格式摩擦。
  6. 已补充连接状态与失败语义：连接中会显示具体目标地址，失败时会区分超时、找不到主机、端口未响应、网络中断等常见情形，便于后续结合 Xcode / 模拟器日志排查。
  7. 已把 OpenClaw 设置页往“首连手输、后续自动回填、始终可编辑”推进一刀：当前会显示已记住的连接目标，并支持一键载入到手动地址模式继续微调，而不是要求用户回到代码里改常量。
  8. 已确认真机“只有 8080 预填”不是设置页绑定缺失，而是设备内并没有持久化过可用主机；当前已新增 app 内 bootstrap 主机兜底，首连时会自动预填默认地址且仍可编辑，但不会把完整 token 硬编码进包里。
  9. 已继续收窄首连 token 摩擦：OpenClaw 设置页新增“粘贴口令 / 清空口令”入口，并明确说明完整 token 首次需从 Mac 复制、之后会本地保存。
  10. 已按测试期策略把 OpenClaw 完整 token 做成 Debug-only 预填：当前真机已验证“主机预填 + token 预填 + 可见清空口令”成立，Release 配置仍不携带该测试 token。
  11. 已补上插件到手机的第一版 host 线索同步：native-plugin 现在会在 `auth-ok` 的 mobile connection profile 中附带 LAN / Tailnet host hints，iOS 会把这些线索并入本地 `OpenClawConnectionProfile`，为后续自动重连和设置页回填打底。
- 当前有效判断：
  1. OpenClaw 原生 Web UI + Tailscale 路线对普通用户偏繁琐。
  2. Thou 若继续保留 OpenClaw 线，唯一值得做的方向是“复杂度压缩器”。
  3. 因此后续重点不再是证明“能不能通信”，而是统一 LAN / Tailnet 的连接心智。
  4. 第一版自动尝试顺序应以“当前网络下最省事、最像本地”的路径优先：先局域网快捷入口，再已知远程地址，最后才退回完全手填。
- 当前待办：
  1. 继续收敛插件导出的连接资料结构，决定第一版哪些 host hints 只作为连接层候选，哪些要显式展示给用户。
  2. 继续把 iOS 的连接入口往“基于连接资料自动尝试”推进，弱化配对码与手动地址两套割裂表单的感觉。
  3. 继续收敛首连体验，并把当前已增强的连接状态/错误反馈与新的 host-hints 连接资料模型打通。
  4. 评估是否要把插件经 `auth-ok` 同步来的连接线索更明显地落到设置页默认可见区，而不只停留在底层自动尝试顺序里。
  5. 决定是否要继续做“本机测试 token 的本地注入方案”；当前主机预填已解决，完整 token 仍保持手填一次后持久化，不默认写入源码或包内配置。
  6. 继续收窄 OpenClaw iOS 偶发卡死的根因；当前已确认 SwiftUI 聊天列表的 lazy layout 是真机 watchdog 崩溃热点之一，`ChatView` 改为非 lazy 堆叠后，用户验证已从“有改善但仍偶发”进一步收敛到“明显改善，基本不再卡死”。
  7. Alice 网络异常的当前判断继续收紧为：VPN/Tailscale 不是必要条件，而是放大器；因为用户已验证在 VPN 连接且 Wi-Fi 开/关两种状态下都出现过成功发送。

- OpenClaw 默认历史读取仍有“旧设备会话链路”残留语义，后续再修，不阻塞当前规划判断。
- OpenClaw UI 仍有状态表达与输入体验细节待收口。
- Alice 的主阻塞已从“archive 是否入链”转为“archive 的启发式提取质量是否足够稳，以及真实运行时是否符合预期”。
- 用户已明确新的产品判断：Alice 与 OpenClaw 的关系不应只理解为“双页并列”，而应理解为“端内上下文工程入口”与“外部 harness 通道”的复杂度分层。
- 因此 Alice 当前的主问题也应从“再补一点功能约束”转向“Methodology A / B 是否足够清楚，能否产出可解释的 retrieve / archive 行为”。
- OpenClaw 工作侧的讨论重心也已变化：后续不应只围绕“静态场景包”规划，而应围绕“推荐工具/技能全集 + 需求分类 + 工作空间动态裁剪”规划。
- 当前终端环境只有 Command Line Tools，没有完整 Xcode，因此本 session 只能完成源码改动与文件级错误检查，不能在这里直接跑 `xcodebuild` 完整验证。
- 当前关于 Alice 的阶段决策已收敛为：先冻结传输层，不把“是否回退请求代码”继续当作主线争论点。

## OpenClaw 配置面补记

- 当前 OpenClaw 不是“把地址和口令重新写死到代码里”的方向。
- 真实源码里已经有独立 `OpenClawSettingsView`，并且 `OpenClawConnectionProfile` 会持久化连接资料。
- 真机排查已确认：把 plist 从设备外部写回并不能可靠改变 Thou 运行时读到的 `UserDefaults` 域，App 启动后会回写自己看到的旧值；因此首连默认值必须走应用内 bootstrap，而不是依赖设备外写偏好文件。
- 当前产品判断已更新：Thou 不再继续默认共用桌面端 `main` 会话；更合理的方向是仍沿用同一个 Thou plugin/channel，但给移动端默认分配 Thou 自己的主会话 key，便于 debug、移动端历史收口与后续产品演进。
- 因此下一步更合理的方向不是回到硬编码，而是：
  1. 首连仍允许用户手动输入或配对。
  2. 连上后尽量通过插件侧快照与 `auth-ok` 同步，把后续常用连接资料自动回填到可编辑设置面。
  3. 在完全没有历史连接资料时，可由 app 内 bootstrap 提供默认主机地址，减少测试期重复手填。
  4. 始终保留用户可修改入口，形态上更接近 Alice 的设置页，而不是隐藏在代码常量里。
