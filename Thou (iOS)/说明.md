# Thou

品牌文化：It’s all about you.

## 当前产品定位

Thou 在 iPhone 上有两条能力线，但当前首发只围绕其中一条交付：

- OpenClaw：工作 companion，负责把 iPhone 接到用户 Mac 上的 OpenClaw，并压缩连接、agent 选择、session 选择与会话读取的复杂度。
- Alice：生活向上下文工程与记忆线，当前保留为后续方向，不进入首发承诺。

## 当前首发判断

- Thou 第一版要和插件一起准备上架，因此首发定义不是“代码能跑”，而是“上架前该准备的产品、工程和分发条件都已就绪”。
- Thou 的首发 UI 只应承接 OpenClaw companion 主线，不应暴露 Alice 页面。
- Thou 不负责重造 OpenClaw 宿主、本地基础设施或远程网络层；它只负责压缩这些能力叠加到手机端时的复杂度。

## 当前阶段定义

### 阶段 1：MVP 收口

定义：Thou 已达到“可以进入上架流程”的状态，但尚未正式发布。

这一阶段必须收口到：

1. iPhone 端能连接 OpenClaw。
2. 能选择 agent、选择 session、读取 history、发送消息。
3. Alice 不进入首发 UI。
4. 真机部署链稳定，签名过期后的恢复路径明确可复现。
5. GitHub 开源与 App Store 上架准备已各自就绪。

### 阶段 2：上架

Thou 的上架面有两个：

1. GitHub 公开仓库。
2. App Store 发布。

其中 GitHub 的开源协议应比插件更严格，但具体协议选择仍以用户最终决定为准。

### 阶段 3：上架后第一阶段

继续降低 OpenClaw companion 的使用摩擦，围绕连接稳定性、agent/session 操作与最小信息架构继续收口。

### 阶段 4：后续阶段

后续继续按当前已有描述推进 Alice、工作流层与更完整的产品关系，但不压进首发承诺。

## 当前协作约定

### 真机验证

- 只要修改了 `Thou/Thou/` 下的 Swift 代码，就必须重新构建并重新部署到真机，旧安装包的表现不能用于判断新修改。
- 真机验证前，必须先确认用户手机上运行的就是本轮刚部署的新包。

### 7 天签名与重新部署

- 当前 Thou 仍采用开发签名直装；手机上“App 不可用”首先应怀疑 7 天签名失效或设备信任状态失效。
- 处理方式不是只看图标是否还在，而是重新走一轮完整部署：
  1. 真机构建成功。
  2. App 成功安装到目标 iPhone。
  3. App 在目标 iPhone 上被实际拉起。
- 若安装后仍无法启动，且系统提示签名、entitlement 或 profile 未被信任，应先在设备端重新确认开发者信任，而不是假设“重新安装即自动恢复”。

### 工具优先级

- 若当前会话可直接使用 XcodeBuildMCP，则优先用它完成 Apple 平台的 build / install / launch。
- 若当前会话未加载或不可达，再回退到本机 `xcodebuild` 与 `xcrun devicectl`。

## 当前边界

- `Reference/` 是 Thou 与插件共用的参考资料库，不是本轮单独交付项目。
- OpenClaw 页当前是外部 harness 的移动端 companion，不是 Thou 内重建一套完整 agent 平台。
- Alice 当前只保留为后续方向，不应继续占据首发文档和首发 UI 的主叙事。

## 当前架构

### App Shell

- `Thou/ContentView.swift` 挂起根导航容器，并把场景生命周期事件交给主聊天状态层。
- `Thou/ViewModels/ChatViewModel.swift` 当前已经收口为单模 OpenClaw shell，只负责页面级状态和与 OpenClaw module 的协调，不再组合 Alice module。
- `Thou/Views/ChatView.swift` 负责首发 UI 外壳：agent inbox、session picker、聊天记录、输入区与设置页。

### OpenClaw Companion 主链

- `Thou/OpenClaw/ViewModels/OpenClawChatViewModel.swift` 负责编排连接、agent 列表、session 列表、history 和 send 主链。
- `Thou/Services/ClawConnectionManager.swift` 是 iPhone 侧 bridge client，负责连接、心跳、重连、历史请求和消息发送。
- `Thou/Models/ChatModels.swift` 提供 OpenClaw companion 所需的轻量聊天模型和连接资料结构。

### 当前工程边界

- 真机验证仍以开发签名直装为主，因此每次修改 Swift 主链后，都应重新 build、install、launch 到目标 iPhone。
- Alice 相关代码资产可以继续保留为后续材料，但不再属于当前公开 App shell 的架构主线。

因此，Alice 不应用“你没有这个需求”来理解用户，而应问：

- 你在用什么方式满足它？
- 为什么你会偏好这种方式？
- 这种方式背后，是性格、资源、时代环境，还是三者叠加？

### 4. 隐藏主线：资源的获取、守护与分配

这份长对话里最稳定的顶层框架之一，是把人的状态理解为对资源的获取、守护与分配。

- 生存与安全，更接近守护资源与应对风险。
- 社交与尊重，更接近分配时间、精力与情绪资源去维系关系和获得位置。
- 认知与审美，更接近把资源投入到精神消费、信息获取与意义体验。
- 真我与实现，则更接近整合已有资源，把它们投向真正重要的方向。

对 Thou 而言，这不意味着 Alice 要把用户强行套进问卷模块，而是意味着 Alice 在理解用户时，可以把“资源如何被投入到不同需求层”当作统一框架。

### 5. 矛盾性是理解深度的来源

Methodology A 不应只追求“总结一致性”，还应主动识别矛盾性。

例如：

- 行为上显得独立，但内里仍强烈渴望连接。
- 口头上谈理想，但实际投入与资源匹配度极低。
- 明明偏好虚拟满足，却又不断渴望真实确认。

这类矛盾不应被当作错误数据，而应被当作人物深度本身。Alice 的价值，不只是说“你是这样的人”，而是帮助用户理解：

- 你为什么会同时拉扯向两个方向。
- 这些选择各自保护了你什么。
- 哪些满足是真实的，哪些只是成本更低的替代。

### 6. 前台语言必须日常化

Methodology A 还包含一条表达原则：理论连接可以在后台完成，但前台语言应尽量日常化。

- 不默认用户懂马斯洛或大五。
- 不直接用术语向用户解释其人格。
- 在真正需要抽象化时，也先用经验语言描述，再决定是否点出理论层。

这条原则对 Thou 同样成立：Alice 的前台表达应像理解你的人，而不是像在进行心理学术语播报。

## Impressions 卡片方向

结合这份长对话，Thou 的 impressions 不应只是静态人格标签，也不应直接等于三我、大五或标签云，而更像“资源-特质-策略-矛盾”组合卡片。

每张卡片至少可包含：

- 当前背景：资源状态或压力来源。
- 核心策略：ta如何满足某类需求。
- 可能代价：这种策略保护了什么，也牺牲了什么。
- 解释语气：用非评判、可共鸣的语言来表述。

因此，impressions 更适合作为：

- 一种人物理解卡片。
- 一种长期可复用的解释节点。
- 一种可被后续 retrieve 拿来支撑叙事的中层材料。

而不只是“高外向、低神经质”这类扁平标签。

它更接近“Alice 目前已经有把握的中层理解”，例如：

- 此人当前的资源地基更像什么。
- ta 常用什么方式满足连接、认知、尊重或实现需求。
- 这种方式通常保护了什么，又牺牲了什么。
- ta 反复出现的拉扯和矛盾在哪里。

因此，impressions 不是整套理论本身，而是 Methodology A / B 作用在具体对话材料上之后，产出的可复用理解材料。

进一步说，这里至少有三层不应混在一起：

### 1. 三我层

长对话后段形成了一个更高层的整合视角：

- 本心之我：由高峰体验和价值观定义，代表“我真正想成为谁”。
- 认知之我：由自我能力评估定义，代表“我认为我是谁”。
- 现实之我：由行动规划和面对困难的态度定义，代表“我现在是谁”。

这套“三我”不是对本我、自我、超我的借名改写，而是长对话中逐步讨论出来的一套新理论。它的目的也不是做经典心理学分型，而是承载 Thou / Alice 想给用户提供的一部分产品价值：

- 帮用户看见这三个“我”。
- 帮用户理解它们为何产生距离。
- 在获得理解后，让三者有机会更和谐地统一。
- 最终落到“了解自己、理解自己、如果可能，成为更好的自己”。

对 Thou 而言，这一层不适合被直接当作每轮都注入的常驻事实，而更适合被理解为：

- 一种高层整合视角。
- 一种在足够材料积累后才逐渐显影的“人物总画像”。
- 一种比单个 topic / impression 更慢、更抽象的解释结构。

因此，三我更接近 Thou 中长期整合出来的高阶人物画像，而不是普通的 topic 条目。

### 2. 大五层

原文后段还明确了一点：大五人格在前台问卷/解读阶段应尽量隐藏在后台，直到所有材料足够完整时，再统一呈现为人格骨架。

把这条判断转译到 Thou 中，大五更适合承担：

- 个性化差异的骨架层。
- 对同一需求满足策略进行个体差异解释的辅助层。
- 一种贯穿多个主题、多个模块、多个时段后才更稳的慢变量画像。

也就是说：

- 资源理论提供的是共性解释底板。
- 大五提供的是个性化风格差异。

因此，Methodology A 不能只有资源地基，还要保留“大五作为个性骨架”的位置，只是这层在前台表达上通常不必术语化。

同时也要避免反过来把大五抬成核心本体：对 Thou 来说，它更像一种成熟、现成、可借用的信息组织方法，而不是产品文化本身。

### 3. 标签云层

原文最后的分享设计里，标签云不是核心理论本体，而是对前面解读结果的压缩索引。

这意味着在 Thou 中，标签云式产物更适合被理解为：

- 对多张 impressions / topics / 三我洞察的压缩表达。
- 一种分享层、摘要层或快速索引层。
- 一种让 retrieve / 展示结果在视觉上更轻、更快被扫读的形式。

因此，标签云不是与 impressions 平级的底层存储对象，而更像由它们二次压缩出的展示层结果。

简化来说：

- impressions / topics：中层可复用解释材料。
- 三我：高层整合画像。
- 大五：个性骨架。
- 标签云：最终压缩索引与展示层。

## Methodology B 草案

Methodology B 对应的不是“如何回答用户”，而是“如何把长对话代谢成可解释的 impressions / topics / archive”。

如果说 Methodology A 负责“Alice 在这一轮应如何看”，那么 Methodology B 更负责“这一轮过后，哪些东西值得留下，以及该以什么粒度留下”。

### 1. 基本流程

当前可提炼的流程为：

- 信号：记录用户说了什么、做了什么、回避了什么。
- 线索：把单个表达转成可解释的倾向、动机或资源信息。
- 整合：跨轮次、跨主题找一致性与矛盾性。
- 卡片：生成稳定的 impressions 候选卡片。
- 叙事：在需要时把卡片重新组织成温和、启发式的解释文本。

### 2. 归档目标不是复读，而是提炼解释材料

Methodology B 的归档，不应只是把聊天压缩成摘要。更重要的是提炼出：

- 用户长期面对什么限制。
- 用户倾向用什么方式满足不同需求。
- 哪些模式是稳定的。
- 哪些拉扯和矛盾反复出现。

如果 archive 只留下“最后一轮在聊什么”，retrieve 时就只能回忆出碎片，而解释不出“为什么会这样”。

### 3. 卡片生成优先级

生成 impressions / cards 时，优先保留：

- 资源地基信息。
- 需求满足策略。
- 反复出现的代价与收益模式。
- 关键矛盾。

不优先保留：

- 单轮情绪噪音。
- 无法复用的表层措辞。
- 没有解释价值的流水事实。

### 4. 表达原则

Methodology B 产出的文本，应遵守：

- 不评判，只呈现与启发。
- 先确认，再深入。
- 优先让用户感到被理解，再让用户看到更深一层。
- 不把卡片直接当结论甩给用户，而是把它们作为解释支架。

这意味着 B 的目标不是生成“正确标签”，而是生成“可被 Alice 后续稳定使用的理解材料”。

### 5. 分层产物原则

结合长对话后半段，Methodology B 产出的对象也应分层，而不是都塞进一个 impressions 桶里：

- 第一层：原始 archive 与结构化 topic / impression 候选。
- 第二层：可复用的人物理解卡片，例如资源地基、满足策略、关键矛盾。
- 第三层：慢变量人格骨架，例如大五倾向。
- 第四层：高层整合画像，例如三我。
- 第五层：分享/展示压缩层，例如标签云、称号、金句。

如果这几层不拆开，后续 retrieve 很容易出现两种坏结果：

- 要么只能取回扁平标签，解释力不足。
- 要么把高度整合的最终画像当作原始事实反复注入，导致过度定型。

因此，Methodology B 也不应把“整份问卷结构”硬塞进归档逻辑里。对 Thou 来说，真正需要借来的不是问卷题目，而是：

- 该优先关注哪些材料。
- 该如何把材料压成中层卡片。
- 什么情况下才允许继续整合成大五或三我。
- 哪些产物只适合展示，不适合进入长期记忆注入。

### 6. 可借用的提示词结构

原文最后形成的一套开发提示词思路，对 Thou 也有直接借鉴价值：

- 先有世界观设定：产品目标、理论主轴、辅助理论、非评判原则。
- 再有角色扮演：AI 以何种语气、立场和边界来理解用户。
- 再有当前任务：当前轮是理解、解读、归档、还是整合。
- 再有输入信息：本轮消息、资源地基、历史卡片、前序模块/前序材料。
- 最后才是步骤化方法：先确认，再呈现，再解释，再启发，再过渡。

对 Thou 来说，这意味着 Methodology A / B 不是一句 system prompt，而更像：

- Master Prompt：定义世界观和总原则。
- Runtime Prompt：定义当前轮如何消费已有材料。
- Archive Prompt / Heuristics：定义本轮如何代谢成下一轮的解释材料。

这套结构虽然来自问卷产品，但对 Thou 的 Alice 同样适用。

如果进一步压到系统提示词颗粒度，当前更合理的做法不是把整套问卷搬进去，而是分成三层：

- Methodology A：少量高层原则，告诉 Alice 当前轮应聚焦资源、需求满足方式、个性差异、关键矛盾。
- Impressions / Topics：少量已经沉淀出的中层理解材料，告诉 Alice 当前这个人有哪些较稳的背景、策略与拉扯。
- Methodology B：少量归档原则，告诉系统一轮结束后什么值得提炼，什么不值得留下。

这样 Alice 既不会因为提示词过厚而失焦，也不会因为没有方向而不知道该围绕什么交流、又该怎样把交流内容变成 impressions 和 topics files。
