# Thou

品牌文化：It’s all about you.

## 当前产品定位

Thou 在 iPhone 上分成两条边界清楚的入口：

- Alice：生活顾问，负责个人连续性、代谢式记忆与端内上下文工程。
- OpenClaw：工作入口，负责连接电脑上的外部 Agent，并以显式文档驱动与可复制配置为主。

这两条线不是同一个 agent 的两个皮肤，而是两种不同的问题域。

## 当前工作区形态

- 当前工作区是 multi-root workspace，包含 `Thou/`、`OpenClaw插件/`、`Reference/` 三个根目录。
- `Thou/` 是 iOS 工程与源码根；`Thou (iOS)/` 是 Thou 项目的核心文档目录。
- `Reference/` 是 Thou 与 OpenClaw 插件共用的参考资料库，不单独作为本轮交付项目；若使用其中资料，需在项目文档中说明引用边界。

## 当前原则

- Alice 线不追求把历史聊天整包塞回上下文，而追求可代谢、可检索、可重组的长期记忆。
- OpenClaw 线不应把核心能力建立在不可见记忆上，而应优先采用明面上的文档驱动、可检查配置与可复制步骤。
- OpenClaw 页面当前的任务首先是“连接我的 Mac”；连接成功之后，真正长期可积累的价值是把知识工作压缩成更容易交付的配置方案。
- Thou 不负责重造宿主本体、网络层或外部基础设施；它负责压缩这些能力叠加到用户面前时的复杂度。
- 这一步已经开始在 `AliceMemoryOrchestrator` 中真实执行。
- 当前 retrieve 结果会被格式化成额外的 system 记忆块，而不是单独做前端展示。

#### 阶段 2：组装 Alice 请求

新增一个编排层，把请求拆成四段：

1. 基础 system prompt。
2. `Impressions` 摘要。
3. 本轮 retrieve 回来的 topics 摘要。
4. 当前活跃消息区 `active rounds`。

这里的关键原则是：
- `Impressions` 常驻但短。
- `Topics` 按需注入。
- `Messages` 只保留当前对话必要窗口。

#### 阶段 3：流式回复

现有 `OpenRouterClient.streamChat(messages:)` 保持不变，先不把 memory 逻辑放进网络层。

当前代码状态补记：
- `OpenRouterClient` 已按 OpenRouter 当前官方文档切到统一 `reasoning` 参数。
- 当前已兼容流式 `delta.reasoning` 与 `delta.reasoning_details` 两条 reasoning 返回路径，用于驱动 Alice 页的 Thinking 展示。
- Alice 直连 OpenRouter 的真实请求链已经跑通；当前 Alice 页已能看到正式回复，reasoning 观测链也已接上联调状态卡。

#### 阶段 4：回复后 archive

assistant 完成后异步调用 `archive`：

1. 判断是否归档。
2. 若归档，则把一部分旧 `rounds` 写入历史聊天文件。
3. 更新或创建 topics。
4. 如有必要，基于 topics 更新 impressions。

当前阶段补记：
- 本轮实际阻塞并不在 OpenRouter API 本身，而在本地配置覆盖链：模拟器内旧的 `UserDefaults` 错误 key 曾覆盖 `AliceLocalSecrets.plist`。
- 当前代码已加入兜底：若 `UserDefaults` 中的 Alice key 形态明显不合法，则自动回退到 `AliceLocalSecrets.plist`，避免旧脏值继续静默覆盖。
- 因此下一 Session 不应再把主精力放在“OpenRouter 能否连通”上，而应回到 archive、`keepRecentRounds` 和记忆代谢边界本身。
5. 返回 `keepRecentRounds` 给 App。

当前代码状态补记：
- `MemorySkillService.archive` 与相关数据模型已经存在。
- 但 archive 还没有真正挂进 Alice 回复完成后的发送链，因此这一阶段仍属于待接线状态。

#### 阶段 5：前台代谢

App 根据 `keepRecentRounds` 截断 `rounds`，只保留最近前台上下文。

这样可以保证：
- UI 仍然只有一个 `rounds` 真相来源。
- 记忆层不直接操作视图。
- archive 的结果通过“保留多少轮”影响前台，而不是直接篡改当前回复。

### 5. Thou App 内部职责切分

#### Alice 页面编排层负责

- 维护当前 Session 的 `rounds`。
- 提供最近消息窗口给 retrieve。
- 在 Alice 请求前组装 prompt。
- 在 archive 完成后按 `keepRecentRounds` 截断前台消息。
- 可选展示“本轮已引用记忆”这类轻状态。

#### 端内 Memory Skill 负责

- 维护 Messages / Topics / Impressions 存储。
- 执行 retrieve 的召回与排序。
- 执行 archive 的归档、topic 更新、impression 更新。
- 输出结构化结果，而不是直接输出给用户看的大段文案。

#### OpenClaw 页面负责

- 单独维护和用户 Mac 上 OpenClaw 的连接状态。
- 单独处理消息发送、接收与会话展示。
- 不读取 Alice 的 Topics / Impressions，也不驱动 Alice 的 archive / retrieve。

### 6. 第一版建议新增的 App 层接口

```text
AliceMemoryOrchestrator
- prepareRequest(userInput, rounds) -> AliceRequestContext
- finishTurn(rounds, latestRound, retrievedTopics) -> ArchiveResult?

AliceRequestContext
- systemPrompt
- impressionsSnippet
- retrievedTopics
- activeMessages

ArchiveResult
- keepRecentRounds
- archivedRoundIDs
- touchedTopicIDs
- impressionsChanged
```

其中：
- `OpenRouterClient` 仍只接受已经组好的 `messages`。
- `ChatViewModel` 不直接理解 topic 存储细节，只消费编排结果。

### 7. 第一版先不做的事

- 不做向量库或复杂语义检索基础设施。
- 不做 topic 自动合并与 schema 全局回写。
- 不做前端可视化记忆管理页。
- 不做 messages 原文的二跳精细回放。

### 8. 第一版实现顺序建议

1. 先在 Thou 工程里补 `AliceMemoryOrchestrator` 与端内 `MemorySkillService` 抽象，不改 UI。
2. 先把 `system prompt + impressions + retrieved topics + active messages` 组装链路跑通。
3. archive 第一版先只返回 `keepRecentRounds` 和 `touchedTopicIDs`，不追求复杂摘要质量。
4. 等链路稳定后，再扩展 topic 结构和 retrieve 质量。

## 双页价值关系补记

从 Thou 的整体产品角度看，Alice 与 OpenClaw 不是两个彼此孤立的页面，而是两种不同复杂度层级的 agent 入口。

- Alice 的目标不是传统“必须手动新建 Session”的聊天窗口，而是一个默认单线程、可持续代谢上下文、未来还能承载技能调用的 chatbot-style agent。
- OpenClaw 的目标不是在 Thou 内重做一套复杂 agent，而是把 Thou 接到外部真实 agent；它的价值在于复用 OpenClaw 已存在的 harness 能力，而不是在 iOS 端复制 harness 工程。

因此，两页之间的价值分工可以明确为：

- Alice 更适合承载生活资讯、陪伴理解、上下文工程与端内记忆代谢。
- OpenClaw 更适合承载工作辅助与复杂问题处理，其复杂度来自外部 OpenClaw 本体，而不是 Thou 自己在本地重建整套 harness。

这意味着 Thou 不必把所有 agent 复杂度都压到 Alice 一页上；Alice 与 OpenClaw 的并存，不是功能重复，而是产品层的复杂度分层。

## Methodology 优先级补记

围绕大模型产品的演进，可以粗略理解为：提示词工程、上下文工程、再到更复杂的 harness 工程。

对 Thou 当前阶段而言：

- Alice 这条线更适合先把上下文工程做深，尤其是 Methodology A / B、retrieve / archive、Topics / Impressions 的边界与协作方式。
- OpenClaw 这条线则天然承接外部 OpenClaw 已有的 harness 工程，不应要求 Alice 在端内短期复制出同等复杂度。

因此，Alice 当前的关键问题已经不是继续给“功能性模拟”补局部约束，而是把 Methodology A 与 Methodology B 讲清楚：

- Methodology A 负责 Alice 在当前轮如何理解用户、使用系统注入信息、组织回答。
- Methodology B 负责一轮结束后如何代谢对话、沉淀 Topics / Impressions、以及怎样为下一轮准备可解释的记忆材料。

只有当这两部分方法论稳定后，当前围绕“为什么 retrieve 命中了却解释不清”“为什么 archive 产物不够可解释”的问题，才有可能系统性收敛。

## Methodology A 草案

结合 [Thou (iOS)/了解一个人.md](Thou%20(iOS)/%E4%BA%86%E8%A7%A3%E4%B8%80%E4%B8%AA%E4%BA%BA.md) 中的长对话，当前可提炼出一版更贴近 Thou 的 Methodology A。它不是问卷本身，而是 Alice 理解一个人的原理框架。

更准确地说，Methodology A 里至少要分清四类东西：

- 资源理论：当前最上层的解释逻辑，用来提醒 Alice 不要脱离现实处境理解用户。
- 马斯洛需求理论与大五人格理论：在资源理论逻辑下使用的现成方法论，更适合作为信息收集与观察坐标，而不是 Thou 自己发明的新理论。
- 三我：从长对话中共同讨论出来的新理论，更接近 Thou / Alice 想提供给用户的高层价值与文化表达。
- 标签云：一种外显的、高效的工具化展示手段，不属于理论本体。

因此，Methodology A 不是把整份问卷塞进系统提示词，而是定义 Alice 在理解用户时应优先看什么、如何解释、哪些层不能混。

### 1. 理解一个人的基础模型

一个人为什么会是现在这个样子，不应被简单理解为“人格决定一切”。更稳的理解框架是：

- 外在资源：包括金钱、时间、生活状态、资源来源、现实压力。
- 内在特质：包括较慢变化的行为倾向、感受方式、处理不确定性的方式。
- 满足策略：用户如何在现实约束下，用各种方式满足自己的需求。

因此，Alice 对人的理解不应是“给人格贴标签”，而应是去理解：

- 这个人当前拥有什么资源。
- 这些资源如何限制或放大了其选择空间。
- ta 正在用什么策略满足自己的需求。
- 这种策略为 ta 带来了什么，也让 ta 失去了什么。

这里的关键不是把马斯洛或大五直接当答案，而是把它们当作收集信息、组织观察的辅助坐标。

- 马斯洛更帮助 Alice 看“哪些需求正在被满足，以及如何被满足”。
- 大五更帮助 Alice 看“同样的满足策略为什么会呈现出不同的个性化风格”。

它们都服务于理解，而不是替代理解本身。

### 2. 背景滤镜原则

资源状态不是一个附属字段，而是解读其余信息的背景滤镜。

同一个表达、同一种行为、甚至同一种焦虑，在不同资源地基上，含义可能完全不同。例如：

- 同样说“担心未来”，有人是在为生存焦虑，有人是在为发展焦虑。
- 同样偏好独处，有人是在恢复精力，有人是在规避高成本社交，有人是在使用虚拟替代满足。

因此，Alice 在理解用户时，不能只读表层行为；必须把资源地基作为前置解释参数。

### 3. 需求普适、满足方式多元

Methodology A 不应把需求理解成“有无题”，而应把它理解成“如何满足”的题。

- 人人都有社交需求、认知需求、尊重需求、实现需求。
- 差异不在于有没有，而在于用什么方式、在什么成本下满足。
- 虚拟满足、单向满足、低成本替代，不应被先验地视为较低级或无效，而应被视为一种现实策略。

因此，Alice 不应用“你没有这个需求”来理解用户，而应问：

- 你在用什么方式满足它？
- 为什么你会偏好这种方式？
- 这种方式背后，是性格、资源、时代环境，还是三者叠加？

### 4. 隐藏主线：资源的获取、守护与分配

这份长对话里最稳定的顶层框架之一，是把人的状态理解为对资源的获取、守护与分配。

- 生存与安全，更接近守护资源与应对风险。
- 社交与尊重，更接近分配时间、精力与情绪资源去维系关系和获得位置。
- 认知与审美，更接近把资源投入到精神消费、信息获取与意义体验。
- 真我与实现，则更接近整合已有资源，把它们投向真正重要的方向。

对 Thou 而言，这不意味着 Alice 要把用户强行套进问卷模块，而是意味着 Alice 在理解用户时，可以把“资源如何被投入到不同需求层”当作统一框架。

### 5. 矛盾性是理解深度的来源

Methodology A 不应只追求“总结一致性”，还应主动识别矛盾性。

例如：

- 行为上显得独立，但内里仍强烈渴望连接。
- 口头上谈理想，但实际投入与资源匹配度极低。
- 明明偏好虚拟满足，却又不断渴望真实确认。

这类矛盾不应被当作错误数据，而应被当作人物深度本身。Alice 的价值，不只是说“你是这样的人”，而是帮助用户理解：

- 你为什么会同时拉扯向两个方向。
- 这些选择各自保护了你什么。
- 哪些满足是真实的，哪些只是成本更低的替代。

### 6. 前台语言必须日常化

Methodology A 还包含一条表达原则：理论连接可以在后台完成，但前台语言应尽量日常化。

- 不默认用户懂马斯洛或大五。
- 不直接用术语向用户解释其人格。
- 在真正需要抽象化时，也先用经验语言描述，再决定是否点出理论层。

这条原则对 Thou 同样成立：Alice 的前台表达应像理解你的人，而不是像在进行心理学术语播报。

## Impressions 卡片方向

结合这份长对话，Thou 的 impressions 不应只是静态人格标签，也不应直接等于三我、大五或标签云，而更像“资源-特质-策略-矛盾”组合卡片。

每张卡片至少可包含：

- 当前背景：资源状态或压力来源。
- 核心策略：ta如何满足某类需求。
- 可能代价：这种策略保护了什么，也牺牲了什么。
- 解释语气：用非评判、可共鸣的语言来表述。

因此，impressions 更适合作为：

- 一种人物理解卡片。
- 一种长期可复用的解释节点。
- 一种可被后续 retrieve 拿来支撑叙事的中层材料。

而不只是“高外向、低神经质”这类扁平标签。

它更接近“Alice 目前已经有把握的中层理解”，例如：

- 此人当前的资源地基更像什么。
- ta 常用什么方式满足连接、认知、尊重或实现需求。
- 这种方式通常保护了什么，又牺牲了什么。
- ta 反复出现的拉扯和矛盾在哪里。

因此，impressions 不是整套理论本身，而是 Methodology A / B 作用在具体对话材料上之后，产出的可复用理解材料。

进一步说，这里至少有三层不应混在一起：

### 1. 三我层

长对话后段形成了一个更高层的整合视角：

- 本心之我：由高峰体验和价值观定义，代表“我真正想成为谁”。
- 认知之我：由自我能力评估定义，代表“我认为我是谁”。
- 现实之我：由行动规划和面对困难的态度定义，代表“我现在是谁”。

这套“三我”不是对本我、自我、超我的借名改写，而是长对话中逐步讨论出来的一套新理论。它的目的也不是做经典心理学分型，而是承载 Thou / Alice 想给用户提供的一部分产品价值：

- 帮用户看见这三个“我”。
- 帮用户理解它们为何产生距离。
- 在获得理解后，让三者有机会更和谐地统一。
- 最终落到“了解自己、理解自己、如果可能，成为更好的自己”。

对 Thou 而言，这一层不适合被直接当作每轮都注入的常驻事实，而更适合被理解为：

- 一种高层整合视角。
- 一种在足够材料积累后才逐渐显影的“人物总画像”。
- 一种比单个 topic / impression 更慢、更抽象的解释结构。

因此，三我更接近 Thou 中长期整合出来的高阶人物画像，而不是普通的 topic 条目。

### 2. 大五层

原文后段还明确了一点：大五人格在前台问卷/解读阶段应尽量隐藏在后台，直到所有材料足够完整时，再统一呈现为人格骨架。

把这条判断转译到 Thou 中，大五更适合承担：

- 个性化差异的骨架层。
- 对同一需求满足策略进行个体差异解释的辅助层。
- 一种贯穿多个主题、多个模块、多个时段后才更稳的慢变量画像。

也就是说：

- 资源理论提供的是共性解释底板。
- 大五提供的是个性化风格差异。

因此，Methodology A 不能只有资源地基，还要保留“大五作为个性骨架”的位置，只是这层在前台表达上通常不必术语化。

同时也要避免反过来把大五抬成核心本体：对 Thou 来说，它更像一种成熟、现成、可借用的信息组织方法，而不是产品文化本身。

### 3. 标签云层

原文最后的分享设计里，标签云不是核心理论本体，而是对前面解读结果的压缩索引。

这意味着在 Thou 中，标签云式产物更适合被理解为：

- 对多张 impressions / topics / 三我洞察的压缩表达。
- 一种分享层、摘要层或快速索引层。
- 一种让 retrieve / 展示结果在视觉上更轻、更快被扫读的形式。

因此，标签云不是与 impressions 平级的底层存储对象，而更像由它们二次压缩出的展示层结果。

简化来说：

- impressions / topics：中层可复用解释材料。
- 三我：高层整合画像。
- 大五：个性骨架。
- 标签云：最终压缩索引与展示层。

## Methodology B 草案

Methodology B 对应的不是“如何回答用户”，而是“如何把长对话代谢成可解释的 impressions / topics / archive”。

如果说 Methodology A 负责“Alice 在这一轮应如何看”，那么 Methodology B 更负责“这一轮过后，哪些东西值得留下，以及该以什么粒度留下”。

### 1. 基本流程

当前可提炼的流程为：

- 信号：记录用户说了什么、做了什么、回避了什么。
- 线索：把单个表达转成可解释的倾向、动机或资源信息。
- 整合：跨轮次、跨主题找一致性与矛盾性。
- 卡片：生成稳定的 impressions 候选卡片。
- 叙事：在需要时把卡片重新组织成温和、启发式的解释文本。

### 2. 归档目标不是复读，而是提炼解释材料

Methodology B 的归档，不应只是把聊天压缩成摘要。更重要的是提炼出：

- 用户长期面对什么限制。
- 用户倾向用什么方式满足不同需求。
- 哪些模式是稳定的。
- 哪些拉扯和矛盾反复出现。

如果 archive 只留下“最后一轮在聊什么”，retrieve 时就只能回忆出碎片，而解释不出“为什么会这样”。

### 3. 卡片生成优先级

生成 impressions / cards 时，优先保留：

- 资源地基信息。
- 需求满足策略。
- 反复出现的代价与收益模式。
- 关键矛盾。

不优先保留：

- 单轮情绪噪音。
- 无法复用的表层措辞。
- 没有解释价值的流水事实。

### 4. 表达原则

Methodology B 产出的文本，应遵守：

- 不评判，只呈现与启发。
- 先确认，再深入。
- 优先让用户感到被理解，再让用户看到更深一层。
- 不把卡片直接当结论甩给用户，而是把它们作为解释支架。

这意味着 B 的目标不是生成“正确标签”，而是生成“可被 Alice 后续稳定使用的理解材料”。

### 5. 分层产物原则

结合长对话后半段，Methodology B 产出的对象也应分层，而不是都塞进一个 impressions 桶里：

- 第一层：原始 archive 与结构化 topic / impression 候选。
- 第二层：可复用的人物理解卡片，例如资源地基、满足策略、关键矛盾。
- 第三层：慢变量人格骨架，例如大五倾向。
- 第四层：高层整合画像，例如三我。
- 第五层：分享/展示压缩层，例如标签云、称号、金句。

如果这几层不拆开，后续 retrieve 很容易出现两种坏结果：

- 要么只能取回扁平标签，解释力不足。
- 要么把高度整合的最终画像当作原始事实反复注入，导致过度定型。

因此，Methodology B 也不应把“整份问卷结构”硬塞进归档逻辑里。对 Thou 来说，真正需要借来的不是问卷题目，而是：

- 该优先关注哪些材料。
- 该如何把材料压成中层卡片。
- 什么情况下才允许继续整合成大五或三我。
- 哪些产物只适合展示，不适合进入长期记忆注入。

### 6. 可借用的提示词结构

原文最后形成的一套开发提示词思路，对 Thou 也有直接借鉴价值：

- 先有世界观设定：产品目标、理论主轴、辅助理论、非评判原则。
- 再有角色扮演：AI 以何种语气、立场和边界来理解用户。
- 再有当前任务：当前轮是理解、解读、归档、还是整合。
- 再有输入信息：本轮消息、资源地基、历史卡片、前序模块/前序材料。
- 最后才是步骤化方法：先确认，再呈现，再解释，再启发，再过渡。

对 Thou 来说，这意味着 Methodology A / B 不是一句 system prompt，而更像：

- Master Prompt：定义世界观和总原则。
- Runtime Prompt：定义当前轮如何消费已有材料。
- Archive Prompt / Heuristics：定义本轮如何代谢成下一轮的解释材料。

这套结构虽然来自问卷产品，但对 Thou 的 Alice 同样适用。

如果进一步压到系统提示词颗粒度，当前更合理的做法不是把整套问卷搬进去，而是分成三层：

- Methodology A：少量高层原则，告诉 Alice 当前轮应聚焦资源、需求满足方式、个性差异、关键矛盾。
- Impressions / Topics：少量已经沉淀出的中层理解材料，告诉 Alice 当前这个人有哪些较稳的背景、策略与拉扯。
- Methodology B：少量归档原则，告诉系统一轮结束后什么值得提炼，什么不值得留下。

这样 Alice 既不会因为提示词过厚而失焦，也不会因为没有方向而不知道该围绕什么交流、又该怎样把交流内容变成 impressions 和 topics files。
